| `GET` | `https://ags.cuzk.cz/arcgis/rest/services/RUIAN/MapServer/exts/GeocodeSOE/tables/1/suggest?text={q}&f=json` | autocomplete suggestion list — minimum trigger ~3 chars |
| `GET` | `https://ags.cuzk.cz/arcgis/rest/services/RUIAN/Vyhledavaci_sluzba_nad_daty_RUIAN/MapServer/1/query?where=adresa%20LIKE%20%27%25{a}%25%27&magicKey={k}&outFields=*&f=json` | full resolution after autocomplete-pick (returns coords + RUIAN codes) |
| `GET` | `https://ags.cuzk.cz/arcgis/rest/services/RUIAN/MapServer/4/query?where=kod={code}&outFields=nazev&f=json` | obec / kraj lookup by RUIAN code |

## Architecture note

This is a **direct browser-to-ČÚZK call**, NOT proxied through SUPIN backend.
Implications:
- User IP hits ČÚZK servers on every keystroke (3+ chars) — privacy/GDPR consideration.
- CSP must allow `ags.cuzk.cz` as `connect-src`.
- ČÚZK rate limits or downtime cascade directly to Bouračka UX.
- No server-side validation of returned addresses; SPA trusts ArcGIS responses verbatim.

## Behaviour observations

- "Václavské" → 15 results in Unhošť, Znojmo, Uhlířské Janovice (alphabetical-ish)
- "Karlovo nám" → 1 result in Polná
- The selected address renders normalised (postcode auto-spaced "273 51" instead of "27351")
- The 3rd query (`MapServer/4/`) resolves a numeric code (`218146`, `173363`) to a
  human-readable obec name; this internal-code-to-name lookup is the boundary
  between ArcGIS-geocoding and the SPA's display logic.

## Failure modes worth asserting

1. **ČÚZK outage** (`ags.cuzk.cz` returns 5xx) → does the SPA degrade gracefully or block submission?
2. **CORS regression** (CSP tightening blocks `connect-src ags.cuzk.cz`) → manual fallback button should still work.
3. **Empty result set** (typed query has no matches) → does the input allow free-text submission?
4. **Slow response** (ČÚZK > 3s) → does the SPA show a loading indicator or hang?
5. **Returned `magicKey` becomes stale** — does the SPA re-suggest on focus?
