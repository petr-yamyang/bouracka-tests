# INT-004 — Vehicle + driver registry (auto-fill)

> Per Template 3 + §5b.3.

```
INTEGRATION ID:    INT-004
INTEGRATION NAME:  Vehicle + driver registry — OCR-read OP/ŘP/SPZ
                   triggers a registry round-trip that auto-fills
                   identity + vehicle data into the wizard
INTEGRATION TYPE:  REST (likely a SUPIN-mediated proxy to public
                   registry; or direct ČKP partner endpoint)
DIRECTION:         outbound (SUT → registry → SUT)
ENDPOINT URL:      unknown (likely internal `/api/registry/lookup`)
WHEN INVOKED:      FLW-003 STEP 5 — after document upload; OCR layer
                   extracts ID number / VIN / SPZ → POST /api/registry
                   /lookup with extracted ids; response carries the
                   auto-fill payload
```

## Request / response shape

Inferred:
- Request — POST `/api/registry/driver-lookup` body `{ id_number, document_type }`.
- Response — `{ first_name, last_name, dob, address, drivers_license_classes, ... }`.
- Vehicle — POST `/api/registry/vehicle-lookup` body `{ vin OR spz }`.
- Response — `{ make, model, year, owner_id, ... }`.

## Timeout behaviour

If registry is slow, the wizard probably surfaces a "Načítání…"
indicator + a manual-entry fallback after a timeout.

## Env-specific divergences

- public — real registry (out of test scope; production-only).
- tst — likely a fixture-backed stub returning canned profiles for
  test SPZ values (e.g. `1ZZ 0001` → "Tester Jeden", DOB
  1990-01-01); critical for deterministic tests.
- tst-demo — DEMO almost certainly uses a fixture stub; auto-fill
  may even be skipped entirely (manual entry) per the "reduced
  validation" posture.

## Test implication

Tests assume the registry stub returns deterministic profiles for
known test SPZ + ID-number values. The mapping table lives in
`fixtures/shared/test-vehicles.json` + `fixtures/shared/test-drivers.json`
and the env config carries the registry-stub URL for SUT diagnostics.
