# INT-002 — SMS gateway (record signing)

> Per Template 3 + §5b.3.

```
INTEGRATION ID:    INT-002
INTEGRATION NAME:  SMS gateway — drivers sign the record by entering
                   an OTP they received via SMS
INTEGRATION TYPE:  REST (likely; SUT → gateway provider) + GSM out (gateway → device)
DIRECTION:         outbound (SUT → SMS provider; both legs)
ENDPOINT URL:      unknown (provider TBC — possibly Smsbrana / SMSManager
                   /Twilio-cs / SUPIN in-house)
WHEN INVOKED:      FLW-003 STEPs 12 + 16 (request OTP for driver A then B);
                   STEPs 14 + 18 (validate OTP entered by user)
```

## Request / response shape

Inferred:
- Request OTP — POST `/api/sms/otp/issue` — body `{ phone: "+420...", channel: "sign" }` → `{ otp_id, expires_at }`.
- Validate OTP — POST `/api/sms/otp/validate` — body `{ otp_id, code }` → `{ valid: true|false, ... }`.

## Timeout behaviour

If the SMS does not arrive in N seconds, the UI exposes "Resend" — number-of-attempts limit observed in FAQ ("Záznam podepíšete jednoduše přes SMS"; FAQ doesn't quote the limit; user template should fill).

## Env-specific divergences

- public — real SMS to real phones; not exercised by automated tests.
- tst — likely **HTTP-based test hook** (read-back OTP) via an
  internal test-only endpoint, e.g. `GET /api/test/sms/last/{phone}`.
- tst-demo — DEMO may use a fixed OTP `0000` always (cheaper than a
  hook + good enough for school-instructor demos).

## Test implication

Strategy:
- Tests in `tst.*` request the OTP via the SUT, then read the OTP
  from the test hook (not from a real GSM device).
- The hook URL + auth is supplied in env config (`env/tst.json`).
- When the hook is unreachable → test fails with a `blocked` verdict
  (not `fail`) — distinguishes infrastructure issues from SUT
  regressions.
