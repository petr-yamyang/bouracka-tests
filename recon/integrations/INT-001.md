# INT-001 — Google reCAPTCHA

> Per Template 3 + §5b.3 data collection points.

```
INTEGRATION ID:    INT-001
INTEGRATION NAME:  Google reCAPTCHA (likely v3 invisible)
INTEGRATION TYPE:  third-party JS + REST (callback siteverify)
DIRECTION:         outbound (browser → google) + outbound (server → google
                    siteverify on submit)
ENDPOINT URL:      https://www.google.com/recaptcha/api.js (script load)
                   https://www.google.com/recaptcha/api/siteverify (server-side)
WHEN INVOKED:      SCR-002 (/formular) page load — script load + token
                   acquisition; FLW-003 STEP 21 — server-side verify on
                   wizard submit
```

## Request / response shape

Script load is a `<script>` include; token acquisition is `grecaptcha.execute(siteKey, { action })` returning a token string. Server-side verify is documented Google API (POST `siteverify` with `secret` + `response`).

## Timeout behaviour

If reCAPTCHA fails to load (third-party blocked), the wizard would
typically submit-block — observed as a stuck "Odeslat" button. Tests
should mock or stub when running in `tst.*` per OQ-CP-14.

## Env-specific divergences

- public — full reCAPTCHA (visible badge confirmed during recon)
- tst — TBC (likely bypassed; awaiting user confirmation per OQ-CP-14)
- tst-demo — TBC (DEMO is exercised many times daily by school
  instructors; bypass is highly likely)

## Test implication

Strategy options (decision per OQ-CP-14):
- **A — Bypass token from SUT.** Simplest; the SUT exposes a magic
  test-mode value when an env flag is set. Tests pass it through.
- **B — Mock at network layer.** Playwright `page.route()` /
  Cypress `cy.intercept()` short-circuits the `siteverify` POST and
  returns a permissive `{ success: true, score: 0.9 }`.
- **C — Real challenge — defer to SUPIN tester.** Used as a sanity
  validation of the dev-stub; tester runs the scenario manually with
  the real reCAPTCHA in tst.* once.
