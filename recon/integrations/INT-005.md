# INT-005 — Maps / geocoder (accident location)

> Per Template 3 + §5b.3.

```
INTEGRATION ID:    INT-005
INTEGRATION NAME:  Map drawing + GPS geocode — user pins the accident
                   location on a map, or sends current GPS coordinates
INTEGRATION TYPE:  third-party JS map (likely Mapy.cz / Google Maps /
                   OpenStreetMap) + outbound REST (geocode/reverse-geocode)
DIRECTION:         outbound (browser → map provider tiles; SUT → geocoder)
ENDPOINT URL:      tiles + reverse-geocode endpoints; provider TBC
WHEN INVOKED:      FLW-003 STEPs 9-10 — accident-location capture
```

## Request / response shape

Inferred — depends on provider. For Mapy.cz: tile fetch + REST
geocoding. For browser geolocation: navigator.geolocation API → no
network until reverse-geocode.

## Timeout behaviour

If the map provider is slow / blocked, the SUT exposes manual-address
fallback (typing an address). This is the deterministic-test path — we
should drive the manual fallback in TC-CP-003 unless the user template
says otherwise.

## Env-specific divergences

- public — real geocoder.
- tst — likely a stubbed geocoder (deterministic lat/lon). The stub
  URL is supplied via env config.
- tst-demo — DEMO may skip the map step entirely (school-instructor
  scenarios often use synthetic locations).

## Test implication

Tests prefer the manual-address path for determinism. The map-pin
path is a separate scenario (deferred until v0.2).
