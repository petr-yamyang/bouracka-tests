# INT-003 — SMTP (record dispatch by e-mail)

> Per Template 3 + §5b.3.

```
INTEGRATION ID:    INT-003
INTEGRATION NAME:  Outbound SMTP — completed record is e-mailed to both
                   drivers with the signed PDF attached
INTEGRATION TYPE:  SMTP outbound from SUT
DIRECTION:         outbound (SUT → SMTP relay → driver inboxes)
ENDPOINT URL:      n/a (SMTP relay; provider TBC)
WHEN INVOKED:      FLW-003 STEP 23 — after submit succeeds, two messages
                   sent (one per driver); subject + body in CS; PDF
                   attachment with the record contents
```

## Request / response shape

SMTP. Test layer doesn't see the wire; tests assert on the
message-capture hook (Mailpit / MailHog / SUPIN equivalent).

## Timeout behaviour

If SMTP fails, the SUT presumably retries N times then marks the
record dispatch-failed. Test hook should expose the dispatch-state
(or the test re-checks after a poll interval).

## Env-specific divergences

- public — real SMTP; unexercised.
- tst — message-capture hook (Mailpit-like); test reads delivered
  messages by recipient.
- tst-demo — DEMO PDFs are watermarked "DEMO ONLY — NEINSTRUKČNÍ"
  (hypothesis per scope C-5).

## Test implication

After submit (FLW-003 STEP 23), poll the message-capture hook for:
- two messages — one to driver-A inbox, one to driver-B inbox
- subject contains the record-id
- body in CS
- single PDF attachment named `zaznam-<id>.pdf` (or similar)

PDF content assertion is OPTIONAL in v0.1 — too brittle. Defer to
v0.2.
