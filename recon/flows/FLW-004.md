## Failure-mode catalogue (each variant = one TC sub-case)

```
F1 — Invalid SPZ format
  TRIGGER POINT: enter SPZ "ABC 123" (does not match Czech pattern)
  ASSERTION:     inline validation message in CS
                 ("Neplatný formát SPZ" or similar)
  CONTROL POINT: form does NOT advance; user remains on the SPZ step
  KIND: trigger + assertion + control.

F2 — Unreadable document photo (OCR fail)
  TRIGGER POINT: upload a deliberately-blurred OP image
  ASSERTION:     UI surfaces "Doklad se nepodařilo načíst, zkuste to prosím
                 znovu" (or similar) and exposes manual-entry fallback
  CONTROL POINT: form continues only after manual override
  KIND: trigger + data_collection + assertion + control.

F3 — Driver-B SMS-OTP mismatch
  TRIGGER POINT: enter wrong OTP for driver B (3 attempts)
  ASSERTION:     each wrong entry shows error; after N attempts the
                 OTP-resend mechanism is triggered (or lockout — env-
                 specific behaviour)
  CONTROL POINT: form does NOT submit until both drivers signed
  KIND: trigger + data_collection + assertion + control.

F4 — Out-of-eligibility location (accident outside ČR)
  TRIGGER POINT: drop GPS pin outside Czech Republic
  ASSERTION:     UI rejects with "Místo nehody musí být na území ČR"
                 or similar; form does NOT advance
  CONTROL POINT: form remains on the location step
  KIND: trigger + assertion + control.

F5 — Damage estimate > 200 000 Kč (eligibility violation)
  TRIGGER POINT: enter damage estimate 250000 (250 000 Kč)
  ASSERTION:     UI re-shows the police-call interlock and offers to
                 redirect; form does NOT submit as a self-record
  CONTROL POINT: regulatory interlock fires.
  KIND: trigger + assertion + control.

F6 — reCAPTCHA challenge fails (or refuses)
  TRIGGER POINT: simulate a reCAPTCHA fail (in tst.* — usually via a
                 test-mode flag that always returns score < threshold)
  ASSERTION:     UI surfaces a non-technical error and offers a retry
  CONTROL POINT: submit blocked
  KIND: trigger + data_collection + assertion + control.

F7 — Server timeout (5xx during submit)
  TRIGGER POINT: drive submit while a network-throttling rule injects
                 a 504 response on POST /api/submit
  ASSERTION:     UI surfaces a recoverable error toast (CS),
                 wizard state is preserved (NOT lost), retry option
                 visible
  CONTROL POINT: form remains submittable on retry.
  KIND: trigger + data_collection + assertion + control.

F8 — Mobile-only: viewport meta missing (regression)
  TRIGGER POINT: open the wizard on a 375 px viewport
  ASSERTION:     viewport meta tag present + initial-scale=1; if NOT,
                 test fails (the wizard would otherwise render zoomed-out)
  CONTROL POINT: smoke + visual.
  KIND: data_collection + assertion.

F9 — Mobile-only: hamburger overlay covers primary submit button
  TRIGGER POINT: open hamburger on the summary step (375 px); attempt
                 to tap submit
  ASSERTION:     submit must remain reachable; if covered → fail
  KIND: trigger + data_collection + assertion.
```

## Env-specific divergences

DEMO's reduced validation may eliminate some failure modes. Specifically
F1 (SPZ format) and F4 (location bound) are likely RELAXED in DEMO. The
TC implementation must mark these as `expectedVerdictPerEnv` —
`{tst: 'fail-shows-msg', tst-demo: 'pass-relaxed'}` — and assert the
two outcomes explicitly per env.

## Integration touchpoints

Same as FLW-003. F2 + F3 + F6 + F7 specifically exercise the
integration-failure assertions documented in INT-001..005.

## Mobile-first viewport spec

Mandatory for F8 and F9; recommended for F1..F7 (all input steps must
work on 320/375/414).
